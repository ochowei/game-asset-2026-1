# Path B — AI-designed bases for medieval-fantasy theme (v1.4.0)

**Status:** Draft, ready for implementation planning
**Date:** 2026-05-06
**Scope:** Single-theme pilot of Path B from `docs/FUTURE-WORK.md`. Implements Story A ("AI in studio, procgen at runtime") with γ architecture (5 bases per subject) for the medieval-fantasy theme only. Other 3 themes remain on v1.3.1 hand-coded primitives.
**Target release:** v1.4.0
**Prior art:** `docs/FUTURE-WORK.md` § Path B (premise, code change shape, brand-reframe risk, acceptance criteria) and § "Open questions to revisit when this work starts".

## 1. Decisions captured during brainstorming

The brainstorming session that produced this spec settled the following questions, all of which were open in `docs/FUTURE-WORK.md`. They are recorded here so the implementation plan does not re-litigate them.

1. **`No AI` brand pillar:** **removed.** AI is permitted in the Procforge authoring pipeline. The runtime pipeline remains AI-free.
2. **Brand pillars going forward:** **Deterministic Runtime / Procedural Variation / Open-source Generator** (replaces `No AI / 100% Procedural / Open-source Generator`).
3. **Story:** **A — Studio/Runtime split.** AI is used at authoring time (committed artifacts only); buyers run a deterministic, offline procgen pipeline. This is the differentiator vs API-based AI icon tools.
4. **Architecture:** **γ.** AI generates N bases per subject; procgen runtime layer applies palette substitution, light transform, and decoration on top.
5. **Scope of v1.4.0:** **Single-theme pilot — medieval-fantasy only** (6 subjects). Other 3 themes (sci-fi, cozy-farm, roguelike-inventory) remain unchanged.
6. **Bases per subject:** **N=5** → 30 base SVGs total for v1.4.0.
7. **Variation richness on bases:** **Moderate.** Palette substitution + light whole-image transform (rotate, scale, conditional mirror). No path-coordinate jitter (would defeat the purpose of designed bases).
8. **AI tool:** Claude (any LLM the author has access to is acceptable; the prompt template is committed for transparency, not the model identity).
9. **Backward compatibility during pilot:** Mixed-model codebase. v1.3.1 hand-coded primitives and v1.4.0 AI-base primitives coexist; `THEME-AUTHORING.md` documents both as valid.

## 2. Architecture

### 2.1 Authoring vs runtime split

The single most important architectural property of Story A is that **AI involvement is bounded to a build-time, committed-artifact authoring step**. The runtime — the code path executed when a buyer runs the CLI or imports `@procforge/core` — has no AI involvement, no API calls, no nondeterministic sampling.

```
AUTHORING (one-time, by author + AI)        RUNTIME (deterministic, in buyer's machine)
─────────────────────────────────────       ──────────────────────────────────────────
Claude generates SVG candidates              generateOne({ theme, seed, size })
         ↓                                            ↓ makeRng(`medieval-fantasy:${seed}`)
author curates                                        ↓ subject composer
(accept / reject / regenerate)                        ↓ pick(rng, primitives) → e.g. swordBlade
         ↓                                            ↓ swordBlade reads imported swordBases
SVG committed to                                      ↓ pick(rng, swordBases) → 1 of 5 SVG strings
_assets/<subject>-<n>.svg                             ↓ palette substitution (token replace)
         ↓                                            ↓ light transform (rotate / scale / mirror)
pnpm build:bases                                      ↓ emit fragment + decoration layer
emits <subject>.bases.ts                              ↓ svgDocument
         ↓
pnpm test + visual QA
```

### 2.2 Determinism contract (preserved)

`(theme.id, seed)` produces byte-identical SVG, **including for medieval-fantasy under the new model**. The contract is preserved because:

- Base SVG strings are constants (committed in `*.bases.ts`).
- All variation in the runtime path comes from RNG draws against the seeded RNG.
- RNG draw order is fixed and documented (see § 4).
- No clock, no filesystem reads (other than committed source), no network.

The determinism baseline for medieval-fantasy resets at v1.4.0 (its byte output changes). Other 3 themes' baselines are unchanged.

### 2.3 RNG draw order for the new variation step

Inside `applyBaseVariation`, the RNG is consumed in this fixed order:

1. `pick(rng, bases)` — selects 1 of 5 base SVG strings
2. `pickColor(rng, palette, 'primary')` — substitutes `{{primary}}`
3. `pickColor(rng, palette, 'secondary')` — substitutes `{{secondary}}`
4. `pickColor(rng, palette, 'accent')` — substitutes `{{accent}}`
5. `pickColor(rng, palette, 'neutral')` — substitutes `{{neutral}}`
6. `range(rng, -8, 8)` — rotation degrees
7. `range(rng, 0.95, 1.05)` — uniform scale factor
8. (mirror-eligible subjects only) `rng() < 0.5` — mirror flag

Steps 2–5 always execute even if the base does not contain that token, to keep the RNG sequence stable across bases that use different role subsets. Each step is exactly one RNG draw producing one hex value; that hex value globally replaces every occurrence of the corresponding `{{role}}` token in the base string. A base using `{{primary}}` three times has all three occurrences resolved to the same hex.

### 2.4 Composer-level RNG order (unchanged)

The `subject` composer (`packages/core/src/composers/subject.ts`) keeps its current RNG draw order: decoration count, decoration positions/angles/picks, then subject primitive call. The new `applyBaseVariation` slot in only consumes RNG from inside the primitive — it does not perturb the composer-level sequence.

## 5. SVG conventions for bases

Base SVGs (in `_assets/`) MUST satisfy:

1. **viewBox:** `0 0 64 64`. No exceptions.
2. **Padding:** silhouette content stays within the inner box `4..60` on both axes (~10% padding). Validated by bounding-box scan over emitted geometry.
3. **Root structure:** single `<g>` element wraps all geometry. No nested `<svg>`. No `<defs>` (no gradients, no clip-paths in base; decoration layer remains free to use them at runtime).
4. **Palette tokens:** colors expressed as one of four placeholder tokens, replaced at runtime:
   - `{{primary}}` — main subject body
   - `{{secondary}}` — secondary surfaces
   - `{{accent}}` — highlights / small details
   - `{{neutral}}` — strokes, dark contrast

   Tokens appear in `fill="{{role}}"` and `stroke="{{role}}"` attributes. A base may use a subset of roles (e.g. only `{{primary}}` and `{{neutral}}`); the runtime still draws all four palette colors per § 2.3 to keep the RNG sequence stable, but unused tokens have no effect.
5. **Stroke widths:** inline as `stroke-width` attribute. Numeric values (no calc, no CSS variables).
6. **No raster:** no `<image>`, no `xlink:href` to external resources.
7. **No scripting:** no `<script>`, no event handlers.
8. **No `<style>` blocks:** styling is inline only. (Reason: simplifies token substitution to a string replace; no CSS cascade to reason about.)
9. **ASCII only:** all path data and attribute values use ASCII characters.

`scripts/build-bases.ts` validates these at build time. A base failing validation aborts the build with a clear error.

## 6. AI workflow + curation discipline

### 6.1 Authoring loop

```
1. Author opens scripts/ai-base-prompt.md, fills in {subject_name} and {subject_description}.
2. Pastes into Claude (or equivalent LLM). Receives N candidate SVG bodies.
3. Author saves candidates to a scratch dir (e.g. /tmp/path-b-candidates/sword/).
4. Author runs `pnpm tsx scripts/qa-base-preview.ts /tmp/path-b-candidates/sword/`
   → outputs PNG renders at 16/32/64/128/256 px, plus a 5-up combined sheet.
5. Author curates: keeps ≥5 acceptable bases, rejects rest, re-prompts as needed.
6. Curated SVGs renamed to <subject>-{1..5}.svg and committed to _assets/.
7. `pnpm build:bases` regenerates *.bases.ts.
8. `pnpm test` runs full suite (including bases-validation.test.ts).
9. Author commits both SVGs and .bases.ts together.
```

### 6.2 Curation rubric (rejection conditions)

A candidate base is **rejected** if any of:

- Silhouette does not read as the named subject at 32 px to a non-author reviewer.
- Any required palette token (`{{primary}}`, `{{secondary}}`, `{{accent}}`, `{{neutral}}`) is missing or wired to the wrong semantic role (e.g. `{{accent}}` used as the main body fill).
- SVG path data is messy: duplicate or zero-area subpaths, off-grid coordinates more than 0.5 unit from integer / half-integer grid, unused `<g>` wrappers, or attribute noise (e.g. inherited stylings on every element).
- Visual similarity to an already-shipped base for the same subject is too high (subjective, but the 5 bases per subject must be visually distinct).
- Violates any rule in § 5.

### 6.3 Re-prompting

If fewer than 5 candidates pass curation, the author re-prompts Claude with explicit feedback (e.g. "previous candidates had handle too thin; produce 3 more with chunkier handles"). This is normal and expected.

### 6.4 Prompt transparency

`scripts/ai-base-prompt.md` is committed to the repo. It contains:

- The SVG conventions from § 5, in prompt-friendly language.
- A per-subject brief slot (filled in at use time).
- Style guidance derived from medieval-fantasy theme tags (`weapon`, `potion`, `shield`, `scroll`, `rpg`, `fantasy`).
- Explicit instruction to produce N candidates per call.

Committing the prompt is part of the brand story: buyers can read what was sent to the LLM. This is the "Open-source generator" pillar applied to the authoring step too.

## 8. Test strategy

### 8.1 Tests modified

- 6 medieval-fantasy primitive tests rewritten as described in § 4.2. Each asserts:
  1. **Base presence:** after replacing each `{{role}}` token in every imported base with its palette-substituted hex value (using the test's seeded RNG to mirror the runtime substitution), at least one of those resolved base strings appears as a substring of the final output.
  2. **Palette substitution complete:** no `{{role}}` token survives in output. All four role tokens checked.
  3. **Transform bounds:** any `transform` attribute parses to rotate within `[-8°, 8°]` and scale within `[0.95, 1.05]`.
  4. **Determinism:** two calls with the same seed produce byte-identical output.

### 8.2 Tests added

- `bases-validation.test.ts` (medieval-fantasy package): For each `_assets/*.svg`, validate the § 5 conventions. Runs as part of `pnpm test`. Catches malformed bases at PR review time.
- `base-variation.test.ts` (core package): Unit tests for `applyBaseVariation`. Inputs: synthetic palette + small synthetic base. Asserts deterministic output, RNG draw order matches § 2.3, transform values within declared bounds.

### 8.3 Tests unchanged

- All other theme tests (sci-fi, cozy-farm, roguelike-inventory) must pass without modification.
- `core` composer tests, RNG tests, palette tests, generator tests must pass without modification.
- `cli` tests, `produce-starter-pack` integration must pass.

### 8.4 Visual QA gate (manual, pre-release)

Before tagging v1.4.0, render 10 random medieval-fantasy seeds at 32 px and 64 px. A non-author reviewer (or the author, on a fresh day) confirms each reads as its named subject. This is the human-judgment gate that automated tests cannot replace. Result documented in the PR description.

## 11. Risks & mitigations

| Risk | Mitigation |
|---|---|
| AI produces SVG that violates § 5 conventions or path data is messy. | `scripts/build-bases.ts` validates at build time; `bases-validation.test.ts` re-validates in test suite. PR cannot land if validation fails. |
| Curation discipline slips → low-quality base lands in `_assets/`. | PR description must include rendered preview of each new base. Reviewer must approve each new base file individually. § 6.2 rubric is referenced in PR template (added in v1.4.0). |
| Mixed-model codebase (medieval-fantasy on v1.4.0 model, others on v1.3.1) confuses contributors. | `THEME-AUTHORING.md` explicit "two authoring paths" section. Each theme's `README.md` (if present) declares which model it uses. |
| Brand reframe undersold in the v1.4.0 listing → itch listing gets pattern-matched to "another AI icon tool". | itch description draft reviewed before publishing. The "Deterministic Runtime" pillar must be the lead bullet, not buried. New tagline tested for clarity. |
| AI output license ambiguity raises buyer concerns. | `docs/AI-AUTHORING.md` § license posture explicitly states the position. README has a one-paragraph summary. Committed prompts demonstrate the bounded scope of AI involvement. |
| Determinism baseline reset breaks downstream consumers who pinned to specific seed outputs. | Documented in CHANGELOG as expected. Rationale for staying on a minor bump (v1.4.0 not v2.0.0): the public API is unchanged, the determinism contract `(theme, seed) → byte-identical` is preserved going forward, and only one of four themes' byte outputs changes. The reset is a one-time event documented in release notes, not an ongoing contract change. Path B's `FUTURE-WORK.md` mention of v2.0.0 was scoped to a full-themes Path B; the single-theme pilot does not warrant a major bump on its own. |
| AI prompt produces inconsistent style across the 6 medieval-fantasy subjects (e.g. sword feels chunky-fantasy, scroll feels minimalist-modern). | `scripts/ai-base-prompt.md` has shared style guidance section reused across all 6 subject briefs. Visual QA gate (§ 8.4) catches inconsistency before release. If detected, re-prompt the outliers. |

## 12. Sequencing for the implementation plan

The implementation plan that follows this spec should be ordered to minimize rework risk:

1. **Pipeline plumbing first** (no AI work yet): `scripts/build-bases.ts`, `scripts/qa-base-preview.ts`, `applyBaseVariation` in core, `bases-validation.test.ts` and `base-variation.test.ts`. Validate end-to-end with a hand-drawn placeholder base for sword-blade only.
2. **Single-subject vertical slice:** drive the pipeline through one full subject (sword-blade) — AI prompt, curate 5 bases, commit, build, test, render. Confirms the loop works before scaling.
3. **Remaining 5 subjects in parallel:** once vertical slice is green, fan out to shield-frame, potion-bottle, scroll-roll, axe-head, gemstone.
4. **Documentation pass:** `AI-AUTHORING.md`, `THEME-AUTHORING.md`, `ARCHITECTURE.md`, `FUTURE-WORK.md` updates.
5. **Brand reframe:** `README.md`, `CHANGELOG.md`, `itch.io` listing draft.
6. **Pre-release:** visual QA gate, full pack regen, prelaunch-check, baseline update if needed.
7. **Tag and ship v1.4.0.**
